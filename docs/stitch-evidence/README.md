# Stitch 设计采集证据说明

## 1. 目录用途

本目录保存 `产销预测智能工作台` 的 Stitch 原始采集证据，用于支撑以下工作：

1. 编写和复核 PRD
2. 回溯原型中的页面事实、文案和状态设计
3. 为研发、测试、设计补充页面级证据
4. 在后续需求讨论中区分“原型事实”和“业务推断”

本目录中的文件是原始设计证据，不是最终需求结论。  
正式的设计反推结论，请查看：

- [2026-06-16-产销预测智能工作台-设计反推PRD.md](../2026-06-16-%E4%BA%A7%E9%94%80%E9%A2%84%E6%B5%8B%E6%99%BA%E8%83%BD%E5%B7%A5%E4%BD%9C%E5%8F%B0-%E8%AE%BE%E8%AE%A1%E5%8F%8D%E6%8E%A8PRD.md)

## 2. 证据来源

本目录内容来自 Stitch 项目在线采集，采集对象为：

- 项目标题：`产销预测智能工作台`
- 选定项目：`projects/4624671007192108229`
- 选定原则：同标题项目中选最近更新时间更晚的版本
- 项目类型：`TEXT_TO_UI_PRO`
- 设备类型：`DESKTOP`

采集范围包括：

1. 项目选择与版本判定信息
2. 项目详情
3. 页面清单
4. 页面详情
5. 设计系统详情
6. 每个页面的 HTML 副本
7. 每个页面的截图副本

## 3. 目录结构

### `json/`

保存 Stitch 返回的结构化原始数据。

- `01-selection-summary.json`
  - 记录同标题项目的匹配结果、选中理由、选定项目
- `02-project-detail.json`
  - 记录选定项目的详情信息
- `03-screens-list.json`
  - 记录页面列表
- `04-screen-details.json`
  - 记录每个页面的详情信息，包括标题、HTML 下载地址、截图下载地址
- `05-design-systems.json`
  - 记录项目的设计系统、样式说明和主题配置

### `html/`

保存每个页面的 HTML 副本。  
用途：

1. 抽取页面文案
2. 查看控件、placeholder、按钮、表格标题
3. 判断页面状态和交互提示

### `screenshots/`

保存每个页面的截图副本。  
用途：

1. 复核页面布局
2. 确认视觉层级、卡片分区和状态展示
3. 辅助理解 HTML 中不直观的结构

## 5. 与 PRD 的关系

PRD 不是直接复制这些文件，而是基于这些文件做了三层整理：

1. `已确认事实`
  - 能直接从 JSON、HTML、截图中证明的内容
2. `合理推断`
  - 基于页面结构、文案和状态关系推导出的产品逻辑
3. `待确认项`
  - 原型无法证明、但研发落地必须明确的问题

如果后续有人质疑 PRD 中某一条结论，应按以下顺序回查：

1. 先查 PRD 对应章节
2. 再查本目录中的 HTML / 截图副本
3. 如仍需更底层证据，再查 `json/` 中的结构化原始数据

## 6. 推荐使用方式

### 对产品经理

- 先读 PRD，再回到这里核对原型事实
- 当需要补充“待确认项”时，优先引用本目录中的页面证据

### 对研发

- 先读 PRD 中的页面需求、状态机和模块说明
- 再用本目录的 HTML / 截图核对页面结构和文案
- 不要直接把截图当成最终交互规范，必须结合 PRD 的事实/推断边界使用

### 对测试

- 可基于页面状态切分测试场景：
  - 登录失败
  - 初始空态
  - 文件解析中
  - 文件解析失败
  - 汇总结果态
  - Agent 对话分析态

## 7. 注意事项

1. 本目录内容是原始设计证据，不代表业务口径已经最终确认。
2. 若后续 Stitch 设计继续演进，本目录可能会过时，需要重新采集。
3. 如果后续重新采集，请保留“采集日期”和“选定项目 ID”，避免不同版本证据混淆。
4. 当前原型中存在已知冲突，使用时要特别注意：
  - 文件大小上限出现 `20MB` 与 `50MB` 两种说法
  - 同标题项目存在两个版本
  - 页面角色文案存在“管理员账号 / 供应链管理主管 / 供应链专家 / 高级供应链专家”多种表述
