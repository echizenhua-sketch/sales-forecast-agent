# 技术评审报告 - 第2部分

## 4. 数据模型设计

### 4.1 核心实体模型

```sql
-- 用户表
CREATE TABLE sys_user (
    id BIGSERIAL PRIMARY KEY,
    username VARCHAR(50) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    real_name VARCHAR(100),
    role VARCHAR(20) NOT NULL,  -- ADMIN, MANAGER, PLANNER, ANALYST
    department VARCHAR(100),
    email VARCHAR(100),
    status VARCHAR(20) DEFAULT 'ACTIVE',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 文件任务表
CREATE TABLE task_file (
    id BIGSERIAL PRIMARY KEY,
    task_name VARCHAR(200),
    file_name VARCHAR(255) NOT NULL,
    file_path VARCHAR(500) NOT NULL,
    file_size BIGINT NOT NULL,
    file_md5 VARCHAR(32),
    upload_user_id BIGINT NOT NULL REFERENCES sys_user(id),
    status VARCHAR(20) NOT NULL,  -- UPLOADING, PARSING, CALCULATING, SUCCESS, FAILED
    error_message TEXT,
    progress INT DEFAULT 0,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- SKU 预测明细表
CREATE TABLE sku_forecast_detail (
    id BIGSERIAL PRIMARY KEY,
    task_id BIGINT NOT NULL REFERENCES task_file(id),
    month VARCHAR(7) NOT NULL,  -- 2024-05
    product_name VARCHAR(200),
    sku_code VARCHAR(100) NOT NULL,
    material_code VARCHAR(100),
    
    -- 库存与排产
    initial_inventory DECIMAL(18, 2),  -- 期初库存
    production_plan DECIMAL(18, 2),    -- 排产计划
    safety_stock DECIMAL(18, 2),       -- 安全库存
    total_supply DECIMAL(18, 2),       -- 总供应
    
    -- SAR 需求组成
    sar_province DECIMAL(18, 2),       -- 省大区 SAR
    sar_dealer DECIMAL(18, 2),         -- 网络经销商 SAR
    sar_export DECIMAL(18, 2),         -- 海外出口 SAR
    sar_internal DECIMAL(18, 2),       -- 内部调拨需求
    sar_total DECIMAL(18, 2),          -- 合计 SAR
    
    -- 缺口与满足率
    gap DECIMAL(18, 2),                -- SAR 差异
    satisfied_demand DECIMAL(18, 2),   -- 可满足合计
    unsatisfied_demand DECIMAL(18, 2), -- 未满足合计
    service_level DECIMAL(5, 2),       -- 满足率 (%)
    
    -- 风险标识
    risk_level VARCHAR(20),            -- CRITICAL, HIGH, MEDIUM, NORMAL, OVERSTOCK, SUFFICIENT
    risk_reason TEXT,
    
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    INDEX idx_task_month (task_id, month),
    INDEX idx_sku (sku_code),
    INDEX idx_risk (risk_level)
);

-- 汇总结果表
CREATE TABLE task_summary (
    id BIGSERIAL PRIMARY KEY,
    task_id BIGINT NOT NULL UNIQUE REFERENCES task_file(id),
    month VARCHAR(7) NOT NULL,
    
    total_supply DECIMAL(18, 2),       -- 总供应
    total_sar DECIMAL(18, 2),          -- 合计 SAR
    total_gap DECIMAL(18, 2),          -- 总缺口
    service_level DECIMAL(5, 2),       -- 满足率
    target_service_level DECIMAL(5, 2) DEFAULT 98.00,  -- 目标满足率
    inventory_turnover_days DECIMAL(5, 1),  -- 库存周转天数
    
    critical_risk_count INT DEFAULT 0,
    high_risk_count INT DEFAULT 0,
    medium_risk_count INT DEFAULT 0,
    
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- AI 会话表
CREATE TABLE ai_conversation (
    id BIGSERIAL PRIMARY KEY,
    task_id BIGINT REFERENCES task_file(id),
    user_id BIGINT NOT NULL REFERENCES sys_user(id),
    session_id VARCHAR(64) NOT NULL,
    question TEXT NOT NULL,
    answer TEXT NOT NULL,
    context JSONB,  -- 上下文信息
    reference JSONB,  -- 引用数据 {"table": "...", "row": 1, "sku": "..."}
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    INDEX idx_session (session_id),
    INDEX idx_task_user (task_id, user_id)
);

-- 操作日志表
CREATE TABLE audit_log (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT REFERENCES sys_user(id),
    username VARCHAR(50),
    operation VARCHAR(50) NOT NULL,  -- LOGIN, UPLOAD, EXPORT, DELETE, etc.
    resource_type VARCHAR(50),       -- TASK, FILE, REPORT, etc.
    resource_id BIGINT,
    detail TEXT,
    ip_address VARCHAR(45),
    user_agent VARCHAR(500),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    
    INDEX idx_user_time (user_id, created_at),
    INDEX idx_operation (operation)
);

-- 导出记录表
CREATE TABLE export_record (
    id BIGSERIAL PRIMARY KEY,
    task_id BIGINT REFERENCES task_file(id),
    user_id BIGINT NOT NULL REFERENCES sys_user(id),
    export_type VARCHAR(20) NOT NULL,  -- SUMMARY, SKU_DETAIL, AI_CONVERSATION
    export_format VARCHAR(10) NOT NULL,  -- XLSX, CSV, PDF
    file_name VARCHAR(255) NOT NULL,
    file_path VARCHAR(500) NOT NULL,
    file_size BIGINT,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

### 4.2 数据模型设计说明

**设计原则**：
1. ✅ 采用关系型数据库设计，支持复杂查询和事务
2. ✅ SKU 明细表包含完整的 SAR 组成，支持可解释性
3. ✅ 汇总表独立存储，提升查询性能
4. ✅ AI 会话记录支持 JSONB，灵活存储上下文和引用
5. ✅ 审计日志独立表，支持合规要求

**索引策略**：
- `task_id + month`：支持按任务和月份快速查询
- `sku_code`：支持 SKU 级查询
- `risk_level`：支持风险筛选
- `session_id`：支持 AI 多轮对话

**扩展性考虑**：
- JSONB 字段支持灵活扩展（AI 上下文、自定义元数据）
- 预留 `department` 字段支持组织架构隔离
- `status` 字段支持任务状态机扩展

### 4.3 数据量评估

假设典型场景：
- 每月上传 10 个任务
- 每个任务平均 5,000 个 SKU
- 保留 1 年历史数据

| 表名 | 单条记录大小 | 年数据量 | 存储空间 |
|------|------------|---------|---------|
| sku_forecast_detail | ~500 字节 | 10 × 5000 × 12 = 600,000 | ~300 MB |
| ai_conversation | ~1 KB | 10,000 对话 | ~10 MB |
| audit_log | ~300 字节 | 100,000 操作 | ~30 MB |
| 其他表 | - | - | ~10 MB |
| **总计** | - | - | **~350 MB/年** |

**结论**：数据量较小，PostgreSQL 完全能胜任，无需分库分表。

## 5. AI 能力实现方案

### 5.1 AI 架构设计

**推荐方案**：混合架构（规则引擎 + 向量检索 + LLM）

```
用户问题
    ↓
┌───────────────────────────────────────┐
│   1. 问题分类与路由                    │
│   - 简单查询：规则引擎直接回答         │
│   - 复杂分析：向量检索 + LLM          │
│   - 超出范围：边界拒答                │
└───────┬───────────────────────────────┘
        │
    ┌───┴────┐
    │        │
┌───▼────┐ ┌─▼─────────────────────────┐
│ 规则   │ │   向量检索 + LLM 生成      │
│ 引擎   │ │                           │
│        │ │   ┌───────────────────┐   │
│ 固定   │ │   │ 1. 向量化问题      │   │
│ 模板   │ │   │ 2. 检索相关数据    │   │
│ 回答   │ │   │ 3. 构建 Prompt     │   │
│        │ │   │ 4. 调用 LLM        │   │
│        │ │   │ 5. 生成引用        │   │
│        │ │   └───────────────────┘   │
└───┬────┘ └─┬─────────────────────────┘
    │        │
    └───┬────┘
        ↓
   回答 + 引用
```

### 5.2 实现方案对比

| 方案 | 优势 | 劣势 | 成本 | 推荐度 |
|------|------|------|------|--------|
| **方案 1：纯规则引擎** | 成本低、响应快、可控 | 灵活性差、扩展性弱 | 极低 | ⭐⭐⭐ |
| **方案 2：规则 + LLM** | 平衡成本和灵活性 | 需维护规则库 | 中 | ⭐⭐⭐⭐⭐ |
| **方案 3：纯 LLM** | 灵活性强、理解能力好 | 成本高、延迟大、可控性弱 | 高 | ⭐⭐⭐ |
| **方案 4：本地开源模型** | 无 API 成本、数据安全 | 部署复杂、效果一般 | 低（硬件成本） | ⭐⭐ |

**推荐方案**：**方案 2 - 规则引擎 + LLM**

### 5.3 AI 成本估算

假设场景：
- 每天 100 个问题
- 平均每个问题 1,500 tokens（输入 1,000 + 输出 500）
- 使用 GPT-4 Turbo（$0.01 / 1K 输入，$0.03 / 1K 输出）

| 项目 | 计算 | 月成本 |
|------|------|--------|
| 规则引擎处理（50%） | 50 问题 × 30 天 = 1,500 问题 | $0 |
| LLM 处理（50%） | 50 × 30 × (1.0 × $0.01 + 0.5 × $0.03) | $60 |
| 向量数据库 | Milvus 自建 / Qdrant 免费版 | $0 |
| **总计** | - | **$60/月** |

**优化建议**：
1. 提升规则引擎覆盖率到 70%，降低成本到 $36/月
2. 采用国内模型（通义千问：¥0.008/1K tokens），成本降低 60%
3. 增加结果缓存，相同问题直接返回

## 6. 关键技术风险与应对

### 6.1 性能风险

| 风险点 | 风险描述 | 影响 | 应对策略 | 优先级 |
|-------|---------|------|---------|--------|
| **大数据量表格渲染** | 10 万行 SKU 数据渲染卡顿 | 高 | 1. 虚拟滚动（react-window）<br>2. 服务端分页<br>3. 默认只加载前 1000 行 | P1 |
| **文件解析耗时** | 50MB Excel 解析超过 30 秒 | 中 | 1. 使用 EasyExcel 流式读取<br>2. 异步处理 + 进度反馈<br>3. 多线程并发解析 | P2 |
| **AI 响应延迟** | LLM 调用超过 10 秒 | 中 | 1. 优先使用规则引擎<br>2. 流式返回（SSE）<br>3. 结果缓存 | P1 |
| **并发上传** | 50 人同时上传文件导致服务崩溃 | 高 | 1. 文件上传限流<br>2. 消息队列异步处理<br>3. 水平扩展解析服务 | P1 |

### 6.2 数据一致性风险

| 风险点 | 风险描述 | 影响 | 应对策略 | 优先级 |
|-------|---------|------|---------|--------|
| **解析失败数据清理** | 解析失败后残留脏数据 | 中 | 1. 数据库事务保证原子性<br>2. 失败自动回滚<br>3. 定时清理任务 | P2 |
| **并发修改冲突** | 多用户同时操作同一任务 | 低 | 1. 乐观锁（version 字段）<br>2. 操作互斥锁 | P2 |

### 6.3 AI 能力风险

| 风险点 | 风险描述 | 影响 | 应对策略 | 优先级 |
|-------|---------|------|---------|--------|
| **LLM API 不稳定** | 第三方 API 故障或限流 | 高 | 1. 主备切换（多个服务商）<br>2. 降级为规则引擎<br>3. 请求重试机制 | P1 |
| **数据引用错误** | AI 引用不存在的数据 | 高 | 1. 引用后验证（查库确认）<br>2. 规则引擎优先<br>3. 明确标注"AI 生成，仅供参考" | P1 |
| **成本失控** | AI 调用量激增导致成本超预算 | 中 | 1. 每用户每日限额<br>2. 缓存常见问题<br>3. 成本监控告警 | P2 |
| **数据安全** | 数据泄露到第三方 LLM | 高 | 1. 脱敏敏感字段（成本、利润）<br>2. 使用私有化部署模型<br>3. 数据加密传输 | P1 |

### 6.4 外部系统集成风险

| 风险点 | 风险描述 | 影响 | 应对策略 | 优先级 |
|-------|---------|------|---------|--------|
| **SAP/WMS 接口不稳定** | 数据同步失败 | 中 | 1. MVP 暂不集成<br>2. 重试 + 降级策略<br>3. 本地缓存备份 | P2 |
| **数据格式不一致** | 外部系统字段变化 | 中 | 1. 字段映射配置化<br>2. 数据校验 + 告警 | P2 |

## 9. 总结与推荐

### 9.1 技术可行性评估

✅ **整体可行**，推荐方案：
- 前端：React + TypeScript + Ant Design
- 后端：Spring Boot + PostgreSQL + Redis
- AI：规则引擎 + 通义千问（LangChain 封装）
- 部署：Docker + Kubernetes（可选）

### 9.2 关键成功因素

1. ✅ **P0 问题必须在 Kickoff 前解决**（核心术语、风险标准）
2. ✅ **MVP 范围克制**（暂不集成 SAP/WMS，降低复杂度）
3. ✅ **AI 能力边界清晰**（规则引擎优先，LLM 补充）
4. ✅ **性能优化前置**（虚拟滚动、异步处理、缓存策略）
5. ✅ **测试覆盖充分**（单元测试 + 集成测试 + 性能测试）

### 9.3 风险提示

⚠️ **中高风险项**：
1. **AI 引用准确性**：需严格验证，避免误导用户
2. **大数据量渲染**：10 万行表格需虚拟滚动优化
3. **第三方 LLM 稳定性**：需主备切换和降级方案
4. **成本控制**：AI 调用需限流和缓存

**✅ 技术方案通过**，建议进入详细设计与开发阶段。
